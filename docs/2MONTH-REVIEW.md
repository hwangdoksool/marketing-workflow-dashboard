## 실제 "마케팅 액션" 유형 분포

2개월간 실제로 일어난 작업을 유형별로 분류:

| 유형 | 건수 | 비중 | 예시 |
|------|------|------|------|
| **웹 구조 변경** | ~12건 | 30% | CTA 강화, 비교표, 결제허들, 섹션 재배치, /links, DNS 전환 |
| **퍼널/시스템** | ~8건 | 20% | 체험 프로세스, 전환 분리, 결제 흐름, 슬롯 관리 |
| **인프라/리포트** | ~10건 | 25% | 데이터 파이프라인, 대시보드, 인텔리전스 리포트, STT |
| **오프라인 소재** | ~3건 | 8% | 리플렛, DP화면, 제안서 |
| **웹 콘텐츠** | ~4건 | 10% | 블로그 팩트체크, 운동 가이드, 시작가이드 |
| **광고 소재** | ~3건 | 7% | UTM 수정, 소재 피로도 대응 (형은님 담당 영역) |

→ **광고 소재는 전체 마케팅 액션의 7%에 불과.** 나머지 93%가 웹 구조/퍼널/인프라/콘텐츠.

## 워크플로우 설계에 반영해야 할 교훈

### 1. "소재"가 아니라 "마케팅 액션" 전체가 루프 대상 (대표님 지시 반영)
- 실제 2개월간 광고 소재보다 웹 구조 변경이 4배 많았음
- Brief 단위 = "가설을 검증하기 위한 모든 행동"

### 2. 문제 발견(Signal)은 마케터의 핵심 역할이어야 하는데, 실제로는 전부 대표님이 했다
- 픽셀 오발화, UTM 미설정, 전환율 병목, 소재 피로도 — 전부 대표님 발견
- **워크플로우가 마케터에게 "데이터를 보고 문제를 찾는 행위"를 구조적으로 요구해야 함**
- Signal 등록 = 마케터의 일상 업무 (AI는 보조)

### 3. 대표님이 직접 실행하는 액션도 루프 안에 있어야 함
- 상세페이지 비교표, 결제허들, CTA — 전부 대표님 기획+지시+검수
- Make 단계에서 리더 전담 가능 (광고 소재만 마케터 자율)
- Brief 작성자 = 마케터 OR 리더 (유형에 따라)

### 4. 피드백 → 수정 → 재피드백 루프의 비효율
- 블로그 초안 피드백 5회 반복 (v1~v5)
- 리플렛 v1~v4 반복
- **Brief 단계에서 방향 합의가 안 되면 Make에서 반복이 불가피**
- Brief 협의 = 시간 투자할 가치 있음 (이후 반복 비용 절감)

### 5. 데이터에서 나온 게 아닌 인사이트도 있다
- 스포엑스 현장: 묠니르 120kg 구경꾼 ≠ 380만원 타겟
- 체험 노쇼: 확정 안내 미발송 → 프로세스 구멍
- 아마존 레퍼런스: "말보다 스펙 자막 + 클로즈업"
- **Signal 유형에 "현장 관찰", "경쟁사/레퍼런스", "프로세스 구멍" 포함 필요**

### 6. 레거시 축적이 안 됐다
- 왜 비교표를 넣었는지, CTA를 바꿨는지 → 기록 없음
- 형은님이 빠지면 3개월 실행 히스토리 전부 증발
- **모든 액션의 "왜"가 Brief에, 결과가 Learn에 남아야 함**

### 7. 주간 리포트의 진화
- 2/15 파이프라인 → 3/9 대시보드 전환 → 3/23 인텔리전스 리포트 → 4/1 워크플로우 대시보드
- 리포트 자체는 계속 발전했지만, **리포트→액션 연결이 약했음**
- Review에서 "이 데이터를 보고 → 다음 Brief로" 흐름이 강제되어야 함
